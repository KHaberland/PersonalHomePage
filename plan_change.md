# План внесения изменений по ТЗ «ФИНАЛЬНАЯ ВЕРСИЯ ГЛАВНОЙ (RU)»

Цель: привести тексты главной страницы к утверждённой редакции на русском, **не меняя** разметку, порядок секций, маршруты кнопок, логику API и компоненты — только содержимое строк (и при необходимости данные в CMS/бэкенде, если они перекрывают переводы).

Основной файл для статического RU: `frontend/messages/ru.json` (секции `home`, для блока опыта при fallback — `experience`). Код страницы: `frontend/app/[locale]/page.tsx`; hero — `frontend/components/Hero.tsx`. Менять код **не требуется**, если достаточно обновить строки и согласованные данные API.

---

## 0. Риск «перекрытия» API (не сломать отображение)

Часть блоков подставляет текст **из бэкенда**, а не из `ru.json`:

| Блок | Когда не хватит правок только в JSON |
|------|--------------------------------------|
| Обо мне (текст) | Если в API заполнено `about.bio_main` — на главной показывается оно, а не `home.aboutText`. Нужно обновить контент в админке/API **или** временно синхронизировать HTML с ТЗ. |
| Ключевые компетенции | Если заполнены `homeTechnicalSkills` / `homeBusinessOutcomes` — карточки и лиды берутся оттуда. Обновить записи в API **или** опираться на fallback из `ru.json` после очистки/совпадения порядка `order` 1…6 и 1…3. |
| Профессиональный опыт | При непустом `getExperience` список с главной строится из API. Fallback из `experience.*.homeOutline` используется только при ошибке/пустом ответе. Для продакшена: привести тексты записей опыта в API в соответствие с ТЗ **или** не полагаться только на JSON. |
| Книга | Заголовок/описание с API `book` перекрывают `home.bookTitle` / `home.bookDescription`. |
| Калькуляторы | Имена и описания с API перекрывают `home.fallbackTool.*`. |
| Блог | Заголовки постов не из ТЗ; ТЗ затрагивает только подзаголовок секции и кнопку — менять только `blogSubtitle` (и при желании единообразие пунктуации). |

**Принцип:** после правок в `ru.json` проверить главную при включённом API так же, как локально, чтобы не осталась старая копия в БД.

## 12. Локали EN / LV и SEO

ТЗ задано для RU. Для **неполомки** мультиязычности:

- После фиксации RU **желательно** обновить `frontend/messages/en.json` и `frontend/messages/lv.json` для тех же ключей (профессиональный перевод), чтобы структура сайта оставалась согласованной.
- Ключи `seo.*` для главной править только если меняется смысл для сниппета (по ТЗ не требуется).

## 13. Порядок работ и проверка

1. Сделать резервную копию или коммит с текущим `ru.json`.
2. Внести правки в `ru.json` (и `experience.*` для fallback опыта).
3. Синхронизировать данные API (about, competencies, experience, book) при необходимости.
4. Локально: `frontend` — сборка/линт; открыть `/ru` (или дефолтную локаль) и пройти все 11 секций глазами.
5. Проверить EN/LV на отсутствие «битых» ключей (после добавления строк в RU все ключи в других локалях должны существовать).
6. Не трогать `page.tsx`, `Hero.tsx`, `HomeSectionProgress.tsx`, роутинг и компоненты карточек — если ТЗ не требует новой вёрстки.
